# Phoenix-Perp — Deep Research Report

> Generated by source-and-docs walkthrough. Describes the system as it exists in the repo today, not the aspirational PRD. Where the two diverge, the code wins and the divergence is flagged.

---

## 1. What this is

`phoenix-perp` (package name `supernova-bot`, product name **SuperNova**) is a **Telegram-native bot for trading perpetual futures on the Phoenix perp DEX on Solana**. Users trade leveraged longs/shorts, manage positions, set TP/SL, get real-time risk/liquidation alerts, monitor other wallets, view a leaderboard, share PnL cards, and earn through a bot-native referral system — all inside Telegram, with a custodial embedded wallet so there are no seed phrases or per-trade signing prompts.

The repo also contains a **second, independent project**: `vulcan-cli/` — a Rust CLI + MCP server for agent-driven trading on the same Phoenix DEX. It shares no code with the bot; both just target Phoenix. (Detailed in §13.)

Key product facts (from PRDs):
- Revenue = **Flight builder fees** (10–15 bps, taker-only), not subscriptions or a token.
- All-in user cost ≈ 4.5–5 bps (Phoenix base 3.5 bps taker + builder fee) + Solana tx fees.
- Target user: Solana "degen" trader, $100K–500K monthly notional.
- US + sanctioned jurisdictions are out of scope (geo-block).
- v1.1 (current MVP) **dropped copy-trading** (deferred to v3, blocked on a Phoenix trader-discovery API) and simplified referrals to bot-native only.

## 3. Process model & startup

Everything runs in **one process** booted from `src/main.ts`. There are no standalone worker scripts — workers are modules with `start*/stop*` exports.

`main()` sequence:
1. `startActionLogRetention()` — immediate sweep + daily `setInterval` deleting `action_logs` older than 30 days.
2. `startAlertWorker()` — BullMQ consumer on the `alerts` queue.
3. `startMarketRefresher()` — refreshes all market snapshots every 25s.
4. `await startWsManager()` — opens Phoenix `traderState` WS subscriptions for every user + enabled monitor.
5. If `NODE_ENV==="production"`: `startLeaderboardScanner()` (non-fatal if it fails).
6. Registers Telegram command list + bot description.
7. **Transport**: production + `WEBHOOK_URL` set → Fastify webhook mode (webhook path = first 32 hex chars of `sha256(bot_token)`, with `WEBHOOK_SECRET` as Telegram secret token). Otherwise → long polling.

Graceful shutdown on SIGTERM/SIGINT stops bot, server, WS manager, market refresher, alert worker, leaderboard scanner. `uncaughtException` triggers shutdown; `unhandledRejection` is logged only.

## 4. Configuration & environment

`src/config/index.ts` validates `process.env` with Zod and **hard-crashes** on failure (field errors in dev, field names only in prod). Notable vars:

- Telegram: `TELEGRAM_BOT_TOKEN`, optional `WEBHOOK_URL`, `WEBHOOK_SECRET` (min 16; required in prod when webhook set — enforced via a `.refine`).
- Privy: `PRIVY_APP_ID`, `PRIVY_APP_SECRET`, **`PRIVY_AUTHORIZATION_PRIVATE_KEY`** (required — this is the server-side signing key for app-owned wallets), optional `PRIVY_AUTHORIZATION_KEY_ID`.
- Phoenix/Flight: `BUILDER_AUTHORITY_PUBKEY` (required), `BUILDER_FEE_BPS` (1–50, default 10), `BUILDER_ACCESS_CODE`.
- Feature flag: `REFERRAL_ENABLED` (default false).
- Phoenix endpoints: `PHOENIX_API_URL` (default `https://perp-api.phoenix.trade`), `PHOENIX_WS_URL` (default `wss://perp-api.phoenix.trade/v1/ws`).
- Solana: `HELIUS_RPC_URL` (required).
- Infra: `DATABASE_URL`, `REDIS_URL`, `PORT` (3000), `HOST` (0.0.0.0).
- **`DEV_SIGNER_SECRET_KEY`** — dev-only base58 64-byte secret key; when set in non-prod, **bypasses Privy entirely** and signs with a local keypair.

Note: CLAUDE.md lists slightly different "required" vars (mentions `PRIVY_APP_ID/SECRET` but the actually-required signing var is `PRIVY_AUTHORIZATION_PRIVATE_KEY`). `BUILDER_FEE_BPS` default is 10 in code.

## 5. Request flow & middleware

```
Telegram → POST /webhook/<slug>  (Fastify, webhookCallback)
  → sequentialize(by from.id)      # serialize a user's updates
  → rateLimitMiddleware            # Redis token-bucket, generic
  → authMiddleware                 # loads ctx.user from DB by telegramId
  → actionLogMiddleware            # wraps command → writes action_logs
  → orderRateLimitMiddleware       # extra cap on /long, /short only
  → command handler
```

- `ctx.user` is `undefined` for new users; `start.ts` handles onboarding, every other command guards with `if (!ctx.user)`.
- A global `bot.on("message:text")` handler in `bot/index.ts` implements **multi-step flows** via Redis "pending" state: key `pending:<telegramId>` holds a colon-delimited token like `trade_size_input:long:BTC`. The handler dispatches on `parts[0]`. Covered flows: withdraw (internal/external/wallet, address entry), deposit amount, settings (custom fee / auto-TP / auto-SL / slippage / leverage), trade size & leverage custom input, price alerts, add-margin, TP/SL price/size/edit, guardian thresholds & margin amounts, wallet-monitor label & add.
- Unmatched callback queries are logged as warnings and acked. `bot.catch` is the global error sink → logs + `renderBotError`.

## 6. Wallet & identity (`src/services/wallet.ts`)

- **Privy app-owned embedded wallets.** `createEmbeddedWallet(telegramUserId)` creates a Privy user with a linked Telegram account, then creates a Solana wallet whose **owner is the app's authorization key** (`getAppPublicKey()` derives an SPKI/DER/base64 pubkey from `PRIVY_AUTHORIZATION_PRIVATE_KEY`). This makes server-side signing and export work without a user JWT.
- `users.telegramId` (string) is the primary identifier and the Privy linked-account id. The DB stores `privyUserId`, `privyWalletId`, `walletAddress`.
- **Signing**: `getPrivyKitSigner(walletAddress)` returns a `SolanaKitSigner`. In non-prod with `DEV_SIGNER_SECRET_KEY` it returns a local keypair signer (bypassing Privy). In prod it resolves `privyWalletId` from DB and builds a Privy `createSolanaKitSigner` with the authorization private key.
- Balance reads use legacy web3.js `Connection`: `getWalletUsdcBalance` (idle USDC in the Privy wallet's token account, standard USDC mint `EPjFWdd5…`) and `getSolBalance`.

**Terminology distinction baked into the product**: "bot wallet" = the Privy Solana wallet (holds idle USDC + SOL for gas). "Trading account" = the Phoenix trader PDA collateral. These are different balances throughout the UX.

## 7. Phoenix integration layer (`src/services/phoenix/`)

### 7.1 Clients (`client.ts`)
Two lazily-created singletons:
- `getPhoenixClient()` — read client (`apiUrl`, `rpcUrl`, no streaming metadata).
- `getTradingClient()` — same plus **Flight routing** when `BUILDER_AUTHORITY_PUBKEY.length >= 43` (a real base58 pubkey). The stub `"111…1"` (32 chars) is treated as "no builder configured" to avoid a proxy panic.

### 7.2 Markets (`market.ts`)
- `getExchangeConfig()` cached 5min; `getMarkets`, `getMarket`.
- `ISOLATED_ONLY_MARKETS = {GOLD, SILVER, SKR, WTIOIL}`; `isIsolatedOnly()` prefers the live `isolatedOnly` flag from exchange config, falls back to the hardcoded set.
- `getMarketSnapshot(symbol)` (cached 30s): mark price (from orderbook mid), tick size, base-lot decimals, **maxLeverage** (= first leverage tier), taker/maker fees, funding rate (latest from funding history /100), open-interest cap, isolated flag, and **leverageTiers** mapped to `{maxLeverage, maxNotionalUsdc}` (tier size × lotToBase × mark).
- `startMarketRefresher()` warms all snapshots every 25s in the background.

### 7.3 Position reads (`position.ts`)
- `getTraderState(wallet)` → normalized `TraderStateEvent`: picks the cross subaccount (index 0) for risk tier / effective & deposited collateral, **aggregates positions across all subaccounts** (cross + isolated), sums unrealized PnL and unsettled funding across subaccounts. Side derived from `virtualQuotePosition` sign; leverage approximated as `positionValue/initialMargin`.
- `getTradeHistory` / `fetchAllTradeHistory` (paginates to 500 fills) and `computeWalletAnalytics` (volume, realized PnL, win count, best/worst trade, per-market breakdown; maker detected via `instructionType === "UncrossCrank"`; a fill counts as a "close" when `realizedPnl !== 0`).

### 7.4 Order execution (`trade.ts`)
The heart of on-chain writes.

- **Fee presets** (Solana priority/Jito tip, not Phoenix fees): `eco` / `normal` / `turbo` map to `{tipLamports, cuPrice}`; `custom` derives both from a SOL amount. Default = `normal` (1.5M lamport tip, 200k µLamport CU price). Compute-unit limit fixed at 250k.
- **Send path**: build v0 tx with [setCUPrice, setCULimit, the instruction(s), Jito SOL tip to a random tip account], sign via the Privy/dev signer, send base64 through **Helius Sender** (`sender.helius-rpc.com/fast`, `skipPreflight:true`, `maxRetries:0`), then `pollConfirmation` (60 attempts × 2s, watches `lastValidBlockHeight` for expiry).
- **Caches/preflight**: SOL balance cached 30s with a 0.005 SOL minimum preflight check; blockhash cached 10s (busted after each send).
- `dispatchInstruction` (single) vs `dispatchInstructions` (batches multiple ix into one tx — used by deposit/withdraw/transfer/TP-SL).
- Public ops: `placeMarketOrder`, `placeLimitOrder`, `closePosition` (ReduceOnly IOC, fraction-aware via `fractionToCloseLots`, abort-on-self-trade), `addMargin`/`depositCollateral`, `withdrawCollateral`, `transferUsdc` (idempotent ATA create + checked transfer), `getUsdcAtaBalanceNative`, and TP/SL set/cancel re-exported from `conditional.ts`.

### 7.5 TP/SL conditional orders (`conditional.ts`)
This is the most intricate module.

- Implements **position-bracket conditional orders** via `buildPlacePositionConditionalOrder`. Ladder-capable: up to **8 active rungs** per market per trader (`MAX_ACTIVE_RUNGS`).
- **Direction mapping**: long TP / short SL → `GreaterThan`; long SL / short TP → `LessThan`. `market` mode = IOC with execution price = trigger ±10% buffer (`computeMarketExecutionTicks`: ×0.9 for Ask close, ×1.1 for Bid close); `limit` mode = limit at trigger price.
- **PDA init is lazy**: `needsConditionalOrdersInit` checks if the conditional-orders PDA account exists (cached in a Set); if not, `buildCreateConditionalOrdersAccount` (capacity 8) is **bundled into the same tx** as the first TP/SL. `markPdaInitialized` records success.
- **Trigger id parsing**: ids look like `ctp-{assetId}-{idx}-{gt|lt}` (TP) / `csl-…` (SL). `parseConditionalId` extracts the conditional-order index + direction used for cancellation.
- Rich **validation**: trigger price must be on the correct side of mark (and SL not at/beyond liquidation); size resolution supports `full | lots | tokens | percent`; total TP and total SL sizes can't exceed position size; multi-rung legs require explicit sizes; final rung count can't exceed 8.
- `buildSetPositionTpSlIxs` assembles: optional PDA-init ix → cancel ixs (for `cancelTpIndices/cancelSlIndices`) → place ixs — all in one atomic batch.
- **Entry point in UX**: `/positions` → tap position → `🎯 Set TP` / `🛑 Set SL` / `🛡 Protect`. There are no `/settp`/`/setsl` commands. Auto TP/SL from settings is applied right after opening a trade (see §9).

### 7.6 Other phoenix files
`lots.ts` (lot/size math: `marginToTokens`, `fractionToCloseLots`), `candles.ts` (OHLC for TA/price), `preflight.ts` (`preflightOpen` — computes notional, effective leverage, fee, liq price, total cost, and raises `PRICE_DRIFT` when an anchor price has moved too far).

## 8. Real-time pipeline (WS → evaluators → alert queue → Telegram)

### 8.1 WS manager (`src/workers/ws.ts`)
- Maintains one Phoenix `traderState` WS **per watched wallet** (`connections` map, cap **500**). Tracks owner wallets (`ownerMap`), external watchers (`watcherIndex`), and an owner→userId LRU cache (max 5000).
- **Reconnect**: exponential backoff (5s × 2^failures, cap 60s, +jitter); after 3 consecutive failures it emits a "live alerts interrupted" alert to the owner and resets the counter.
- **On each `traderState` event** (`handleTraderStateEvent`): reads previous positions from Redis (`ws:positions:<wallet>`, 1h TTL), then for the **owner** runs, in order:
  1. `evaluatePositionFlip` — detects direction flips (invalidates TP/SL) and other position-change alerts.
  2. `evaluateRiskTier` — risk/liquidation tier alerts.
  3. `evaluateGuardianRules` — user-defined risk automation.
  4. Clears guardian "peak" tracking for positions that disappeared.
  5. If `REFERRAL_ENABLED`: accrues referral fees on fills (`accrueReferralFee` per fill notional).
  For **external watchers** (non-owner), runs `evaluateMonitorAlerts`.
  Finally persists current positions back to Redis.
- **Dynamic subscriptions** via Redis pub/sub channel `monitor:events` (`subscribe`/`unsubscribe` with wallet+telegramId), so wallet-monitor add/remove from the bot process reaches the WS manager. Also starts the price-alert watcher.

### 8.2 Evaluators (`src/workers/evaluators/`)
- `risk-tier.ts` — maps Phoenix risk tiers (`atRisk/at_risk`, `cancellable`, `liquidatable`, `backstopLiquidatable`, `highRisk`) to escalating messages (⚠️ Margin Low → 🟠 Warning → 🚨 Liquidation Risk → 🆘 Imminent). Built with `FormattedString`. Inline keyboard offers fast de-risk actions (Add margin / Add stop) when exactly one position is open. Respects per-user alert toggles and dedups on `ws:dedup:<tid>:risk:<type>` (300s).
- `guardian.ts` — the rules engine evaluator (see §10).
- `position-flip.ts`, `monitor.ts`, `price-alert.ts` (with `startPriceAlertWatcher` polling), `shared.ts` (`isAlertEnabled`, `esc` HTML-escape for legacy raw-HTML alerts).

### 8.3 Alert queue & worker
- `jobs/queues.ts`: BullMQ `alerts` queue, 3 attempts, exponential backoff, keep last 100 completed / 500 failed. `AlertJobData = {telegramId, type, message, entities?, symbol?, keyboard?}`.
- `jobs/processors/alert.ts`: concurrency 10. **Dedup** via Redis `SET NX EX 5` on `alert:dedup:<tid>:<type>:<symbol>` (5s window). Sends with `entities` when present, else falls back to `parse_mode:"HTML"` (legacy callers). Non-retryable `BotError`s are dropped (not retried); retryable ones rethrow so BullMQ retries.

## 9. Trade flow UX (`src/bot/commands/long.ts`, mirrored by `short.ts`)

Size-first redesigned flow: **symbol → size (margin) → leverage → confirm → execute**.

- `/long` (and `/short`) gate on `ctx.user` and `phoenixActivated`. One-liner form `/long BTC 10x 500` jumps straight to confirm; `2.5x` decimal leverage accepted.
- **Symbol picker** is paginated (10/page) and renders each market as a Telegram **deep link** (`t.me/<bot>?start=<side>_<SYMBOL>_<page>`) showing price, max leverage, funding. Isolated-only markets show `[ISO]` and are blocked at the size step ("not available yet").
- **Size step** reads effective collateral; offers preset buttons + custom entry (sets pending `trade_size_input:…`).
- **Leverage step** validates against `maxSafeMargin = available / (1 + maxLev × totalFeeRate)` (so fees fit at max leverage), shows the default with a ★, and each button shows resulting exposure. With `confirmTrades=false` ("instant mode"), picking leverage fires the trade immediately.
- **Confirm** runs `preflightOpen` and shows a detailed card: margin vs exposure, entry, fee + %, total cost, **max slippage**, funding daily cost/income, **liq price with distance**, and a **≥20× high-leverage warning**. The anchor price is embedded as `toFixed(12)` (avoids scientific notation; fixes an earlier `PRICE_DRIFT` bug). Buttons: Open / Change size / Change leverage / Refresh price / Cancel.
- **Execute** (`executeTrade`):
  - Re-runs preflight (with anchor) → on `PRICE_DRIFT` shows inline Refresh instead of restarting.
  - Acquires a Redis trade lock (`trade:lock:<userId>`, 150s) to prevent concurrent trades.
  - Idempotency: confirm callback claims `claimIdempotencyKey(from.id, callbackQuery.id)`.
  - Edits message to `CONFIRMING`, then runs the on-chain order in a detached IIFE (so the webhook handler returns fast). On success: records trade (`recordTrade`), subscribes the wallet to WS, optionally sets **auto TP/SL** from settings (TP=limit, SL=market, after a 2s indexer delay; failure is non-fatal with a warning), and edits the message to a success card with a `🛡 Protect` button. On error: edits to a `txError` card with Retry/Back.

## 10. Guardian (risk-rule automation)

A user-configurable rules engine — the bot's headline safety feature.

- Schema (`guardian` table): `ruleType ∈ {liq_distance, drawdown, pnl_target, funding_drain, exposure_limit, margin_ratio}`, optional `symbol` (null = portfolio-wide), `threshold`, `direction`, `action ∈ {notify, suggest, auto_close, auto_reduce, auto_margin}`, `actionParam`, `enabled`, `cooldownSec` (default 300), `lastTriggeredAt`.
- Service (`services/guardian.ts`): CRUD with **30s in-memory active-rules cache** (invalidated on mutate), max 20 rules/user, `disableAllAutoActions` (downgrades all auto-* to `suggest` — used by `/guardian off`).
- Evaluator (`evaluators/guardian.ts`) runs on every owner `traderState` event:
  - Honors per-rule cooldown.
  - **liq_distance**: % distance mark→liq below threshold.
  - **drawdown**: tracks peak PnL per symbol in Redis (`guardian:peak:…`); fires when drop from peak ≥ threshold%.
  - **pnl_target**: PnL as % of estimated margin vs threshold (direction above/below).
  - **funding_drain**: estimated daily funding cost ≥ threshold.
  - **exposure_limit** / **margin_ratio**: portfolio-level (work even with the position filter empty).
  - On trigger: `markTriggered`, then either queue a **suggest/notify alert** (with Close/Reduce-50%/Add-$100/Snooze inline buttons) or **execute an auto action** (`auto_close`/`auto_reduce`/`auto_margin`) under a 150s Redis lock, sending a receipt (with Solscan link) or a failure alert with manual-close buttons.

## 11. Other commands & features

`src/bot/commands/` (registered via `index.ts`): `start` (onboarding), `activate` (Phoenix activation via builder access code), `wallet`, `deposit`, `withdraw`, `portfolio`, `positions`, `markets`, `long`, `short`, `tpsl`, `history`, `funding`, `leaderboard`, `share` (PnL card), `referral`, `claim`, `guardian`, `alerts`, `settings`, `pricealert`, `wallet-monitor`, `log`, `status`, `help`.

## 12. Leaderboard / trader discovery (`workers/leaderboard.ts` + `services/leaderboard.ts`)

Production-only. A 5-phase startup then two recurring cycles:
1. Sync wallet tags from `data/wallet-tags.json`.
2. **GPA discovery** — `getProgramAccounts` on the Phoenix program filtered by the TRADER discriminator; parses collateral / market count / last-update-slot from account binary.
3. Seed `leaderboard_snapshots` (500-row batches), hydrate changed traders via Phoenix REST.
4. Hydrate bot users (activated wallets).
5. **WS `trades` subscription** per market — each new `taker` is deduped in Redis (`lb:known:<wallet>`, 1h) and `upsertAndHydrateWallet`'d.

Recurring: **30-min** stale-trader backfill; **2-hour** full GPA rescan + history backfill. A shared `TokenBucket(2, 0.5)` rate-limiter keeps it ≈1–2 req/s (Phoenix 429'd around 5 req/s). `getLeaderboard` sorts by total volume / win rate / realized PnL with pagination. Snapshots carry both `discoveredVia` `gpa` and `ws_trades`.

## 13. `vulcan-cli/` — the nested Rust project

A **separate, independent** AI-native CLI + MCP server for trading Phoenix perps. Shares no code with the bot; only the DEX.

- **Workspace**: two crates — `vulcan` (clap binary) and `vulcan-lib` (logic, MCP server, wallet, paper engine, indicators, strategy runners, error handling).
- **Deps**: Phoenix Rise SDK (Rust, v0.1.3), Solana SDK ~2.3, `rmcp` (MCP), `kand` (TA indicators), `aes-gcm`+`argon2`+`zeroize` (encrypted local wallets), Tokio.
- **Command groups** (~16): setup, status, market, trade, position, margin, account, portfolio, paper, strategy (TWAP/grid/TA), ta, history, wallet, auth, agent, mcp.
- **Skills system**: ~18 `skills/*/SKILL.md` markdown playbooks (entry-point `vulcan` skill + focused skills for trade execution, risk, TP/SL, TWAP, grid, scale orders, TA strategy, portfolio, margin, onboarding, etc.). Loaded by agents progressively to save tokens. Tool/error catalogs are machine-generated and CI-checked.
- **Safety model**: paper + dry-run always safe; MCP server read-only by default; live signing requires `--allow-dangerous` on the server *and* `acknowledged:true` per call; wallet unlock via env at startup; all dangerous ops logged to `~/.vulcan/agent-log.json`; private-key export is user-only (agents forbidden).
- **Integration**: installs skills + registers an MCP server into Claude Code / Cursor / Codex. Tools named `vulcan_<group>_<action>`.

It looks like a parallel/experimental product (agentic trading) living in the same repo as the consumer Telegram bot.

## 14. Conventions & standards (enforced)

- **Message formatting**: always `@grammyjs/parse-mode` (`fmt` tagged template + `FormattedString`), send `{entities}` not `parse_mode:"HTML"`. Raw-HTML alert callers are legacy and must `esc()` every interpolation. (Alert worker still falls back to HTML when no entities present.)
- **On-chain tx copy** comes from `src/bot/lib/tx-flow.ts` (`SUBMITTING`, `CONFIRMING`, `txSuccess`, `txError`, `TX_MSG_OPTS`) — never hand-rolled.
- **Errors** all route through `BotError` / `toBotError` / `renderBotError` (`src/bot/lib/errors.ts`). Taxonomy: 10 categories, ~21 codes; SDK error strings are pattern-matched to friendly messages + hints + retryable flag. Retryable: slippage, blockhash-expired, rate-limit, network. Non-retryable: insufficient SOL/margin, not registered, no position, isolated-only.
- **Terminology** locked in `src/bot/lib/terms.ts`: "bot wallet", "trading account", "margin", "position size", "liquidation price".
- **UX standards** in `docs/ux-standards.md` + `docs/ux-flows.md` (number formatting, buttons "← Back" / "✕ Cancel" / "✅ <Verb>" / "View on Solscan →", ≥20× leverage warning, bracket-first "🛡 Protect" TP/SL).
- **ESM rule**: `.js` import extensions everywhere; no `require()`.
- **Communication style** (CLAUDE.md): caveman — terse, bullets, no filler.

Key protocol facts encoded in the product:
- Phoenix USDC (`PhUsd…`) ≠ standard USDC (`EPjFWdd5…`); deposits/withdrawals wrap 1:1 through the **Ember proxy**.
- Account PDA = `(wallet_authority, portfolio_index, subaccount_index)`; subaccount 0 = cross-margin, >0 = isolated. GOLD/SILVER/SKR/WTIOIL require isolated.
- Effective collateral = deposited + discounted positive uPnL + negative uPnL + unsettled funding; risk tiers escalate Low→AtRisk→Cancellable→Liquidatable→BackstopLiquidatable→HighRisk.
- Funding paid every 8h, settled daily; mark price (not last trade) drives PnL.
- Withdrawals go through a global FIFO queue (budget ~2M Phoenix USDC, ~450/slot replenish).

## 17. Notable design decisions & observations

1. **Single-process monolith** with in-memory maps (WS connections, caches) — simple to run, but state (owner/watcher maps, PDA-init set, snapshot caches) is **not shared across instances**, so horizontal scaling would need rework. The Redis `monitor:events` pub/sub hints at an intended multi-process split that isn't fully realized.
2. **Custodial signing** via Privy app-owned wallets is the core trust/risk surface. The dev `DEV_SIGNER_SECRET_KEY` bypass is gated to non-prod. The PRD calls for KMS for the authorization key — the code reads it straight from env.
3. **Fast tx path**: Helius Sender + Jito tips + `skipPreflight` + custom confirmation polling, with blockhash/SOL caches to cut RPC latency. CU limit is a fixed 250k.
4. **Atomic multi-ix batching** (`dispatchInstructions`) for deposit/withdraw/transfer and especially TP/SL (PDA-init + cancels + places in one tx).
5. **Detached execution** of the on-chain order in `executeTrade` keeps the webhook responsive; the message is later edited via stored chatId/msgId.
6. **Defensive caching everywhere** with TTLs (exchange 5min, snapshot 30s, SOL 30s, blockhash 10s, owner LRU 5000, guardian rules 30s) and rate-limiting (generic + order-specific middleware, leaderboard token bucket).
7. **WS connection cap of 500** is a hard scaling ceiling for live alerts per process.
8. CLAUDE.md "Known bugs (all verified fixed)" list matches code (alert toggle filter, InputFile for photos, T2 referral filter, ws deps).

### Bottom line
A well-structured, single-process TypeScript Telegram bot that wraps the Phoenix/Rise perp DEX with a custodial Privy wallet, a size-first trade flow, ladderable position-bracket TP/SL, a real-time WS alert + risk-guardian pipeline, a referral/leaderboard growth layer, and image-based PnL sharing. The same repo separately hosts `vulcan-cli`, a Rust agentic-trading toolkit for the same DEX. Main risk surfaces: custodial key handling, Rise/Flight SDK beta status, and the in-memory single-process scaling ceiling.
