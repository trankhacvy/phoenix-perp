④ Close Position Flow

close:SYM:PCT:side → optional confirm → executeClose() ▼ collapse
Stage 1 — Callback: close:SYM:PCT:side
close:SOL:100:long → ctx.user? → getSettings(user.id) → settings.confirmClose?
confirmClose = false
checkOrderRateLimit(ctx) → ≤5 orders/min?
Over limit → answerCallbackQuery "Too many orders"
OK → claimIdempotencyKey(userId, callbackId) → NX claim ok?
Already claimed → return (double-submit blocked)
Claimed → executeClose(ctx, sym, pct, side, fee)
confirmClose = true
getTraderState(wallet) → pos found?
Found → compute: closingSize, closingUsdc, estimatedPnl, remainingSize
reply confirm message → buttons: [✅ Close X] [✕ Cancel]
Confirm message shows: symbol, side, close %, mark price, size being closed (in tokens + USD), estimated P&L excl fees, and remaining size if partial.
Stage 2 — Confirm callback: close:exec:SYM:PCT:side
close:exec:SOL:100:long → answerCallbackQuery("Closing…") → checkOrderRateLimit → claimIdempotencyKey → executeClose()
Stage 3 — executeClose() (runs async in background)
editMessageText("⏳ Closing position…") → closePosition(sym, wallet, fraction, fee)
closePosition() internals:
1. getTraderStateSnapshot(wallet) — fetch raw lots from subaccounts
2. Compute: rawLots = pos.basePositionLots, isLong = rawLots > 0
3. fractionToCloseLots(rawLots, fraction) — size in base lots
4. Build IOC ReduceOnly order packet (bid if short, ask if long)
5. dispatchInstruction(ix, wallet, fee):
  a. getBlockhash() (cached 10s)
  b. Build v0 transaction: CU price + CU limit + close ix + Jito tip
  c. sendViaHeliusSender(base64Tx)
  d. pollConfirmation(sig) — polls every 2s, up to 60 attempts, checks slot vs lastValidBlockHeight
On-chain success
getTraderState(wallet) → pos still in state?
Found → recordTrade() → fire-and-forget, no await
edit: ✅ "Position closed" + Solscan link
fraction = 1 (full close) → buttons: [Long][Short] / [Positions]
fraction < 1 (partial) → buttons: [View position] / [Positions]
generatePnlCard(trade data) → api.sendPhoto(chatId, pnlCard.png)
PnL card generation uses satori + resvg. Wrapped in try/catch — if card fails, success message is already displayed. PnL % = unrealizedPnl * fraction / margin * 100.
On-chain error
toBotError(e) → build error message → edit: ❌ "Close position failed" + hint + retry?
Error message includes: userMessage, hint (if any), "↩️ Safe to retry." (if retryable). Button: [← Back → nav:positions].
Cancel button on confirm screen
cancel → answerCallbackQuery() + editMessageText("Cancelled.")
Generic cancel callback — registered globally, not in positions.ts. Just dismisses the confirm message.
Edge cases summary
ScenarioResult
User not authenticatedReturns early silently (ctx.user guard)
Global rate limit exceeded (>20/min)Middleware returns "Too many requests" before handler
Order rate limit exceeded (>5 orders/min)Handler returns "Too many orders. Wait a minute."
Double tap (same callbackId within 120s)Idempotency NX check returns false → silently discarded
Position not found (race — closed externally)confirmClose=true: no position block in confirm msg. executeClose will throw "No open position" from SDK
closePosition SDK throwsError displayed in message, "← Back" button shown
Tx confirmed on-chain but already expiredpollConfirmation throws "Transaction expired before confirmation"
editMessageText fails after successCaught + logged as warn — tx is already done
PnL card generation failsCaught + logged as error — success message already shown

⑤ Add Margin Flow

margin:SYM → free-text amount → addmargin:exec:SYM:AMT ▼ collapse
Stage 1 — Prompt: margin:SYM callback
margin:SOL → ctx.user? → getTraderState(wallet)
pos found for symbol?
Not found → reply "No open SOL position. It may have been closed."
Found → build prompt message
reply: "Add Margin — SOL" prompt → setPending(userId, "addmargin:SOL")
Prompt shows: Available collateral (effectiveCollateral) and current liq price. Pending key TTL = 600s (10 min).
Note: position lookup uses .find(p => p.symbol === symbol) — no side filter. Finds first position in that market regardless of long/short.
Stage 2 — Free-text amount input (message:text handler)
user types "50" → getPending(userId) = "addmargin:SOL"
parseAmount(text) → NaN or < 1?
Invalid → reply "Invalid amount. Enter $1 or more:" + [✕ Cancel]
Valid → clearPending(userId)
reply: "Add $50.00 margin to SOL?" + [✅ Add $50.00][✕ Cancel]
Stage 3 — Execute: addmargin:exec:SOL:50 callback
addmargin:exec:SOL:50 → answerCallbackQuery("Adding…") → checkOrderRateLimit
claimIdempotencyKey(userId, callbackId) → editMessageText("⏳ Adding margin…")
getSettings(user.id) → getFeeConfig(feeMode, customFeeSol) → addMargin(sym, wallet, amount, fee)
addMargin() = depositCollateral(wallet, amountNative, fee)
amountNative = BigInt(Math.round(amountUsdc * 1_000_000))
Uses buildDepositIxs({ authority, amount, traderPdaIndex: 0 })
Then dispatchInstructions(ixs, wallet, fee) — may be multiple ixs
Stage 3 — Result paths
✓ Success
edit: "✅ Added $50.00 margin to SOL."
✗ Error
toBotError(e) → edit: "❌ Add margin failed" + hint + retry?
editMessageText also wrapped in try/catch — if edit fails, logged as warn.
Async pattern: Both success and error paths run inside an IIFE (async () => { ... })() that the callback returns immediately after starting. This ensures the Telegram API doesn't time out while waiting for the tx.

RATE LIMITS
  Global:     ratelimit:{userId}         →  20 req/min  (middleware, blocks before handler)
  Orders:     ratelimit:orders:{userId}  →   5 req/min  (handler-level, checked in close/addmargin)

IDEMPOTENCY
  Key:  idem:{userId}:{callbackId}   TTL=120s   Redis NX
  Prevents duplicate tx if user double-taps the same inline button

PENDING STATE
  Key:  pending:{userId}   TTL=600s
  Set:  margin:SOL prompt  →  addmargin:SOL
  Used: message:text handler routes to correct sub-flow

ASYNC EXECUTION
  Both executeClose() and addMargin execute() run as async IIFEs after the
  callback handler returns. This avoids Telegram's 10s callback timeout.
  Errors inside the IIFE are logged but don't propagate to bot.catch().

FEE CONFIG
  eco:    600K lamports tip  +  100K μlamports CU price
  normal: 1.5M lamports tip  +  200K μlamports CU price   (default)
  turbo:  7.5M lamports tip  +  1M   μlamports CU price
  custom: user-specified SOL amount → converted proportionally

Regex note: Symbol pattern [A-Z0-9]+ matches uppercase letters and digits only. Symbols like WTIOIL, BTC, SOL all match. Side is strictly long or short (lowercase). Mismatches fall through to the generic callback_query:data handler which logs a warning.

⑨ Redis State & Guards Summary

All Redis keys touched by position detail flows ▼ collapse
Redis Key TTL Type Set by Read by Purpose
ratelimit:{userId} 60s (rolling) INCR + EXPIRE NX rateLimitMiddleware rateLimitMiddleware Global 20 req/min cap
ratelimit:orders:{userId} 60s (rolling) INCR + EXPIRE NX orderRateLimitMiddleware, checkOrderRateLimit checkOrderRateLimit Order-specific 5 req/min cap

pending:{userId} 600s SET EX setPending() in margin: handler getPending() in message:text Routes free-text to correct multi-step flow
Key insight: The pending key for margin input is "addmargin:SOL". The message:text handler splits on ":" and checks parts[0] === "addmargin" then reads parts[1] as the symbol. Invalid input does NOT clear the pending key — user can keep trying until TTL expires or they hit Cancel.
